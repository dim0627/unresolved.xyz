ぼんやりしたタイトルやテーマを起点に、対話で記事のプロットを詰めて、そのまま本文ドラフトの書き下ろしまで行う。`grill-me` と同じインタビュー形式で構成を固めたあと、Markdown 本文を `apps/blog/content/posts/<slug>.md` に保存する。

## 流れ

1. **入口確認** — 起動時にユーザーから出ているタイトル候補・テーマ・伝えたい一点を確認する。タイトルが仮なら「仮タイトル」のまま進め、最終確定は後段で行う
2. **対話で詰める** — 下記の判断分岐を一つずつ解消する。質問は一度に一つ、各質問には推奨回答案を添える（ユーザーが「それで」と言えば即採用）
   - 想定読者（誰に届けたいか、その人は何を知っていて何を知らないか）
   - 記事のゴール（読み終えた読者にどう変わってほしいか・何を持ち帰ってほしいか）
   - 種別（技術メモ・TIL / 設計や思想を語る読み物 / 書評・ポエム / 振り返り 等）
   - 章立ての骨格（導入 → 本論 → まとめ 等の構造、章数の目安）
   - 各章の要点（一行で）
   - キーメッセージ（記事を一文で言うと何か）
   - 触れない範囲（スコープ外の明示）
   - 既存記事との関係（参照したい記事、続編か、差別化ポイント）
3. **プロット仮確定** — 上記を「## プロット段階のテンプレート」の形式にまとめて応答に提示し、ユーザー確認を取る。修正があればその場で反映
4. **タイトル確定** — 仮タイトルなら、詰めた内容を踏まえてタイトル案を3つ提示してユーザーに選んでもらう
5. **slug 決定** — 確定タイトルから kebab-case の英数字スラッグ案を2-3個提示。`apps/blog/content/posts/` 配下の既存ファイル名と衝突しないか確認した上でユーザーに選んでもらう
6. **本文書き下ろし** — プロットを「## 本文ドラフトのスタイルガイド」に従って本文化する。各章の要点メモを段落に展開し、既存記事（特に同種別のもの）のトーンに揃える。ユーザーが事前に「プロット段階で止めて」と明示した場合のみスキップ
7. **ファイル保存** — `apps/blog/content/posts/<slug>.md` に書き出し、保存先パスと「次やること」を一行で添えて報告する

## ルール

- 質問は一度に一つ。`grill-me` と同じ流儀
- ユーザーが既に答えている内容や、明示済みの方針については再質問しない
- 各質問には推奨回答案を必ず添える
- 既存記事（`apps/blog/content/posts/*.md`）の文体・章立て・タグの傾向を参考にしてよい。必要に応じて読み込む
- タグは既存タグ（TIL / TypeScript / JavaScript / React / Next.js / AI / Claude Code / 個人開発 など）から候補を提示する。新規タグも提案可だが、既存と意味が被るものは寄せる
- 外部資料の参照（WebSearch / WebFetch / Notion 等）はユーザーが明示的に指示した場合のみ。デフォルトは「ユーザーから情報を受け取る」立場
- デフォルトは本文書き下ろしまで行う。プロット段階で保存して別途自分で書きたい場合は、ユーザーが明示的に指示する

## プロット段階のテンプレート

プロット仮確定フェーズで応答に提示する形式。各章は1-2文のリード文＋要点メモのハイブリッドにすることで、本文化したときのイメージが浮かびやすくする。ファイル保存ではなく応答内で見せるための形式なので、コードフェンスで囲って提示する。

````
---
title: "<タイトル>"
date: "<YYYY-MM-DDT00:00+09:00>"
tags:
  - "<tag1>"
  - "<tag2>"
---

想定読者: <ひと言で>
キーメッセージ: <ひと言で>
スコープ外: <ひと言で>

書き出し: <記事の入り口で何を提示するか、1-2文の語り口>

## <章1のタイトル>

リード: <章1の冒頭にくる1-2文の語り口>
要点:
- <要点メモ>
- <要点メモ>

## まとめ

リード: <まとめの語り口>
要点:
- <持ち帰ってほしい一点>
````

## 本文ドラフトのスタイルガイド

最終的に `apps/blog/content/posts/<slug>.md` として保存される本文を書くときの方針。

- **frontmatter**: `title` は二重引用符付き、`date` は ISO 8601（+09:00 オフセット、起動日）、`tags` はリスト形式で各要素を二重引用符付き
- **frontmatter コメント**: プロット段階で記載した想定読者・キーメッセージ等のメタ情報コメントは、本文化したファイルには残さない（既存記事の慣習に合わせる）
- **基調**: 既存記事のトーン（です・ます調、私見エッセイ寄り）に揃える。`apps/blog/content/posts/llm-era.md` などをサンプルとして観察する
- **文体**: 種別が「思想・読み物（poem系）」のときは「〜と思います」「〜のではないでしょうか」「〜という気がします」などの私見ニュアンスを基本とする。技術メモ系はそれよりも事実ベースの記述で書いてよい
- **段落構成**: 各章2-4段落程度。プロット段階の要点メモを1段落=1ポイントとして展開する
- **箇条書きの使い方**: 本文中の箇条書きは「並列の要素を列挙したいとき」など必要な箇所に留め、文章で表現できるものは段落にする
- **Table of contents**: 既存記事に揃えて、導入直後に `## Table of contents` を空セクションとして置く

## 既存記事を参考にしたいとき

- `apps/blog/content/posts/` 配下から、近いトピックの記事を1-2本サンプリングして文体・章構成・タグを観察する
- 観察したものを次の章立て提案や文体方針に反映する。引き写しはしない
